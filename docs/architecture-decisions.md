

## ADR-001: Выбран вариант доменных микросервисов

### Контекст

Система должна поддерживать домены:

- пользователи;
- каталог;
- персонализированная лента;
- заказы;
- платежи;
- уведомления.

Дополнительно нужно явно показать границы ответственности, владение данными и C4 Container.

### Варианты

1. Доменные микросервисы (`User`, `Catalog`, `Feed`, `Order`, `Payment`, `Notification`) + `API Gateway` + `Message Broker`.
2. Модульный монолит с внутренними модулями и отдельными интеграционными адаптерами.
3. Макросервисы по бизнес-потокам (`Identity`, `Merchant`, `Commerce Core`, `Engagement`) + event-stream + read-модели (CQRS-подход для чтения).

### Решение

Принят вариант 1: доменные микросервисы.

### Причины выбора

- наиболее явно демонстрирует границы доменов;
- лучше соответствует требованию отсутствия shared DB;
- позволяет разделить синхронные и асинхронные связи на уровне контейнеров;
- естественно масштабируется по профилю нагрузки (например, `Feed` отдельно от `Order`).
- минимизирует методологические риски для учебного этапа при сохранении архитектурной полноты.

### Почему не варианты 2 и 3

Вариант 2 (модульный монолит) не выбран, потому что:

- в меньшей степени демонстрирует границы сервисов именно на Container-уровне C4;
- ограничивает независимое масштабирование доменов при росте нагрузки.

Вариант 3 (макросервисы + event-stream) не выбран, потому что:

- требует более сложного управления событийными контрактами и read-моделями;
- на учебном объеме создает избыточную сложность относительно требований задания;
- укрупняет домены внутри сервисов, снижая прозрачность ownership.

### Последствия и trade-off

Плюсы:

- независимое масштабирование и релизы;
- прозрачная ответственность команд/сервисов;
- лучшее соответствие DDD-границам.

Минусы:

- выше операционная сложность (наблюдаемость, CI/CD, контракты);
- сложнее сквозная отладка распределенных сценариев;
- больше накладных расходов на сетевое взаимодействие.

## ADR-002: Единая точка входа через API Gateway

### Решение

Внешние клиенты обращаются только в `API Gateway`; прямой доступ к внутренним сервисам не предполагается.

### Причины выбора

- единая точка для маршрутизации;
- единый слой для аутентификации/авторизации и лимитирования;
- упрощение внешнего API и версионирования.

### Последствия и trade-off

Плюсы:

- унификация клиентских интеграций;
- проще внедрять кросс-сервисные политики.

Минусы:

- риск бутылочного горлышка;
- необходимость высокой доступности Gateway.

Митигирующие меры:

- горизонтальное масштабирование;
- health-check и readiness/liveness проверки;
- timeouts, retries и circuit breaker на downstream вызовах.

## ADR-003: Коммуникации: синхронные запросы + асинхронные события

### Контекст

Часть операций требует мгновенного ответа клиенту, а часть удобнее обрабатывать с низкой связанностью между сервисами.

### Решение

Использовать смешанную модель:

- синхронно: клиентские запросы через Gateway, оркестрационные вызовы между сервисами;
- асинхронно: доменные события через `Message Broker`.

### События

- `OrderCreated`, `OrderStatusChanged`;
- `PaymentSucceeded`, `PaymentFailed`;
- `ProductUpdated`.

### Последствия и trade-off

Плюсы:

- уменьшение временной связанности сервисов;
- проще организовать ретраи и повторную доставку;
- удобная интеграция `Notification Service` и обновлений `Feed Service`.

Минусы:

- eventual consistency;
- необходимость идемпотентных потребителей;
- усложнение трассировки бизнес-потока.

## ADR-004: Границы владения данными без shared DB

### Контекст

Требование задания: явно определить границы владения данными и исключить разделяемые базы между сервисами.

### Решение

У каждого сервиса собственное хранилище данных.

Распределение владения:

- `User Identity Service` -> `users`, `roles`, `credentials`, `sessions`;
- `Catalog Service` -> `products`, `prices`, `stock`, `categories`;
- `Feed Service` -> `ranking_features`, `recommendation_cache`;
- `Order Service` -> `orders`, `order_items`, `order_status_history`;
- `Payment Service` -> `payments`, `payment_attempts`, `payment_status_history`;
- `Notification Service` -> `notification_templates`, `delivery_log`, `delivery_status`.

### Последствия и trade-off

Плюсы:

- четкая ответственность за данные;
- снижение связности между сервисами на уровне БД;
- проще эволюционировать схемы по отдельности.

Минусы:

- усложнение межсервисных выборок и аналитики;
- потребность в проекциях/read-моделях для агрегированных отчетов.

## ADR-005: Модель персонализации ленты на этапе проектирования

### Контекст

Нужно обеспечить персонализированную выдачу, но без реализации бизнес-логики в рамках текущего этапа.

### Решение

Зафиксирована гибридная концепция `Feed Service`:

- rule-based фильтрация и базовое ранжирование;
- учет пользовательских сигналов;
- fallback для cold start (популярные товары по сегменту/категориям).

### Последствия

Плюсы:

- понятный путь эволюции от простой к более сложной персонализации;
- можно начать с минимальных данных и улучшать качество постепенно.

Риски:

- без качества сигналов ранжирование может быть слабым;
- потребуется контроль дрейфа модели/правил на следующих этапах.

## ADR-006: Объем реализации на этапе "инициализация сервиса"

### Решение

Реализован только `User Identity Service` как технический каркас:

- код: `cmd/useridentity/main.go`;
- тесты: `cmd/useridentity/main_test.go`;
- контейнеризация: `Dockerfile`, `docker-compose.yml`;
- endpoint: `GET /health`.

Остальные сервисы зафиксированы архитектурно в C4 и ADR, но не реализованы кодом на текущем этапе.

### Последствия

- требования задания соблюдены;
- архитектура описана полностью;
- технический риск минимален за счет малого объема реализации.
