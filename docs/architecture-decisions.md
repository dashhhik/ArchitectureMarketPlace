# Почему я выбрала такую архитектуру

## ADR-001: Разбила систему по доменам
- Решение: сделала отдельные сервисы для доменов (Пользователи, Каталог, Лента, Заказы, Платежи, Уведомления).
- Почему: так проще держать четкие границы и независимо масштабировать те части, которые реально нагружаются.
- Цена решения: больше межсервисного взаимодействия и больше инфраструктурной работы.

## ADR-002: Наружу только через API Gateway
- Решение: клиенты ходят только в Gateway, а не в сервисы напрямую.
- Почему: в одном месте удобнее держать маршрутизацию, auth и лимиты.
- Цена решения: Gateway становится критичной точкой, поэтому его нужно делать надежным.

## ADR-003: События через брокер
- Решение: статусы по заказам и платежам передаются через брокер сообщений.
- Почему: сервисы меньше зависят друг от друга, а ретраи и повторная обработка делаются проще.
- Цена решения: консистентность становится не мгновенной, и обработчики должны быть идемпотентными.

## ADR-004: Без общей базы данных
- Решение: у каждого сервиса своя база (или своя схема владения данными).
- Почему: четко видно, кто отвечает за какие данные, и меньше связность на уровне хранилища.
- Цена решения: для общей аналитики нужны проекции, read-модели или репликация.
