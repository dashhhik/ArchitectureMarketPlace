# Маркетплейс: ДЗ по архитектуре (C4 + запуск сервиса)

## 1. Что тут сделано
Я собрала учебный каркас маркетплейса под это ДЗ.

Что закрыто в работе:
- разбиение системы на сервисы
- C4 Container-диаграмма
- один рабочий сервис в Docker
- health-check, который возвращает `200 OK`

## 2. C4 Container-диаграмма
- картинка: `docs/c4-container.png`

На диаграмме есть:
- акторы: `Покупатель`, `Продавец`
- внешние системы: `Платежная система`, `Провайдер Email/SMS/Push`
- контейнеры маркетплейса: `API Gateway`, сервисы доменов и `Брокер сообщений`
- отдельная БД для каждого доменного сервиса

## 3. Домены и кто за что отвечает
1. `Пользователи и идентификация`
- регистрация и профили
- роли и вход в систему
- базовые данные по пользователю

2. `Каталог`
- товары и карточки
- цены, атрибуты, остатки
- изменение каталога со стороны продавца

3. `Лента и рекомендации`
- персональная лента
- ранжирование товаров
- выдача релевантных позиций

4. `Заказы`
- оформление заказа
- смена статусов
- хранение состава заказа

5. `Платежи`
- запуск оплаты
- фиксация результата
- связь с внешней платежной системой

6. `Уведомления`
- отправка сообщений по событиям
- хранение истории отправок
- доставка через внешний провайдер

## 4. Как я разложила домены по сервисам
Идея простая: **один домен = один сервис**.

- домен пользователей -> сервис пользователей
- домен каталога -> сервис каталога
- домен ленты -> сервис ленты
- домен заказов -> сервис заказов
- домен платежей -> сервис платежей
- домен уведомлений -> сервис уведомлений

Почему так:
- можно масштабировать сервисы отдельно (например, ленту)
- понятнее ответственность команд
- изменения в одном сервисе меньше бьют по другим

## 5. Данные и границы владения
Общей БД нет. У каждого сервиса своя база, и это специально, чтобы не смешивать ответственность.

1. `Сервис пользователей`
- данные: пользователи, роли, креды, сессии
- зона ответственности: авторизация и идентификация

2. `Сервис каталога`
- данные: товары, категории, цены
- зона ответственности: актуальность каталога

3. `Сервис ленты`
- данные: признаки персонализации, кэш выдачи
- зона ответственности: рекомендации

4. `Сервис заказов`
- данные: заказы и статусы
- зона ответственности: жизненный цикл заказа

5. `Сервис платежей`
- данные: платежные транзакции и статусы
- зона ответственности: работа с платежным провайдером

6. `Сервис уведомлений`
- данные: шаблоны и история отправок
- зона ответственности: доставка уведомлений

## 6. Как сервисы общаются
1. Синхронно:
- `Клиент -> API Gateway` (HTTPS)
- `API Gateway -> сервисы` (HTTP внутри)
- `Сервис заказов -> Сервис платежей` (gRPC/HTTP)
- `Сервис платежей -> внешняя платежка` (HTTPS)

2. Асинхронно:
- `Сервис заказов -> Брокер`: `OrderStatusChanged`
- `Сервис платежей -> Брокер`: `PaymentConfirmed`, `PaymentFailed`
- `Сервис уведомлений <- Брокер`: читает события и отправляет сообщения

## 7. Два варианта декомпозиции
### Вариант A (я выбрала его)
Доменные микросервисы: отдельный сервис на каждый ключевой домен.

### Вариант B
Модульный монолит + отдельные интеграционные адаптеры на платежи и уведомления.

## 8. Плюсы и минусы вариантов
### Вариант A
Плюсы:
- гибкое масштабирование
- независимые релизы
- понятные границы доменов

Минусы:
- сложнее поддерживать инфраструктуру
- сложнее отлаживать межсервисные сценарии
- обычно дороже в эксплуатации

### Вариант B
Плюсы:
- проще стартовать
- проще локально отлаживать сквозные сценарии
- меньше накладных расходов на старте

Минусы:
- хуже независимое масштабирование
- со временем модули могут сильно сцепиться
- потом сложнее переходить в полноценную распределенную архитектуру

## 9. Почему я выбрала вариант A
Для этого кейса он лучше показывает, что именно проверяют в ДЗ:
- явные границы доменов
- владение данными без shared DB
- событийные взаимодействия (заказы/платежи)
- нормальный путь к масштабированию, если проект вырастет

## 10. Что реально реализовано в коде
Я реализовала Сервис пользователей и идентификации как минимальный технический каркас:
- стек: `Go (net/http)`
- endpoint: `GET /health`
- ответ: `200 OK` + JSON
- бизнес-логика отсутствует

Код: `cmd/useridentity/main.go`
